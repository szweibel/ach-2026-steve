  • Media tools for transcription, OCR, and image description
  • An AI Use Disclosure tool that records what you delegated

The Tools Portal is where the lab puts standalone apps, separate from the chat Sandbox Zach showed you. Some handle media — transcription in dozens of languages, OCR for archival scans, alt-text for accessibility. There's a disclosure tool that gives a scholar a citable record of which tasks they handed to a model. The ones I want to dwell on are the agentic studios. All of it runs on zero-retention paths.

Working with agentic AI

  • Agent Studio and Site Studio run agents for research and publishing
  • People use them for research and scholarly-communication work
  • It takes several steps to finish a task on its own
  • Faculty and students build working tools with little code
The studios are how we put agentic AI in people's hands — not a chatbot you ask, but an agent you hand a task to. That shift, from asking to delegating, is what's reshaping research workflows and scholarly communication. Someone can build a small research tool or an academic site without writing every line, and learn what it means to work alongside an agent.

Kale Workbench

governed coding-agent work

  • A browser workbench where a coding agent does the building
  • The agent only uses tools the lab has reviewed
  • Code runs in an on-demand sandbox, under budget caps and an audit trail
  • You approve any push or secret, then it hands off to Kale Deploy
Workbench is our browser environment for governed coding-agent work. You work with a coding agent, but it can only call tools the lab has reviewed, and any code it runs happens in a sandbox that spins up on demand and is destroyed after. The whole session is under CUNY policy — sign-in, budget caps, an audit trail — and the agent can't push to GitHub or use a secret without your approval. When the project is ready, Workbench hands it off to Kale Deploy. (Active repo, roadmap maturity.)

Kale Deploy

publish to a CUNY address

  • You push the repo, or hand off from Workbench, and it goes live at <project>.cuny.qzz.io
  • It works the way Vercel or Netlify do, with the lab running it
  • The whole thing runs on Cloudflare Workers, and CUNY holds the credentials and policy
  • You see on the commit whether it worked
Once something's built, Kale Deploy publishes it. A push, or a handoff from Workbench, and the project is live at a CUNY address. It works the way Vercel or Netlify do, except the lab operates it, on Cloudflare Workers, rather than each person renting space from a company. The institution holds the credentials and the policy. (Do not call this on-prem — it runs on Cloudflare.)
